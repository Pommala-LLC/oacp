### 1. Message-kind taxonomy review

**Status:** Open for v0.2 review.
**Origin:** Draft-00 audit.
**Locked decision for v0.1:** Defer; do not rename.

#### Background

The v0.1 draft uses `messageKind` values such as `EVENT`, `RESPONSE`, `PROPOSAL`, `TOOL_CALL`, `TOOL_RESULT`, `GATE_RESULT`, `REVIEW`, `BINDING`, `CAPABILITY_NEGOTIATION`, `ESCALATION`, `RUN`, `SYSTEM` (and the v0.2-direction `TOOL_REQUEST` and `ERROR` already accepted in the schema enum).

The draft-00 audit proposed an alternate taxonomy in which `TOOL_CALL_REQUESTED` would be modeled as `messageKind = TOOL_REQUEST` and `messageType = TOOL_CALL_REQUESTED`, with `ERROR` as a first-class message kind. The proposed canonical set:

```
COMMAND, EVENT, QUERY, RESPONSE, ERROR, ESCALATION, REVIEW,
TOOL_REQUEST, TOOL_RESULT, GATE_REQUEST, GATE_RESULT,
EVIDENCE, PROPOSAL, CONTROL, SYSTEM
```

#### Why deferred

A mid-flight rename would touch the spec body (especially §10 broker state machine), all conformance vectors, all examples, all bindings, and the schemas. The current taxonomy is internally consistent. The cost-benefit didn't favor a v0.1 rename.

#### V0.2 review questions

1. Is the broader kind/type split (`TOOL_REQUEST` as kind, `TOOL_CALL_REQUESTED` as type) cleaner than the v0.1 collapse (`TOOL_CALL` as both kind and prefix)?
2. Should `ERROR` be a first-class message kind?
3. Should `COMMAND` and `QUERY` be added (currently absent from v0.1)?
4. How does any rename interact with existing `messageType` strings already in production envelopes?

#### Migration path if approved

- Add new kinds to the enum without removing old ones.
- Mark old kinds `DEPRECATED` per spec §27.7.
- Update spec body, vectors, examples, bindings.
- 90-day deprecation window before old kinds become `RETIRED`.

### 2. Proposal lifecycle naming review

**Status:** Open for v0.2 review.
**Origin:** Draft-00 audit (second pass).
**Locked decision for v0.1:** Defer; spec body retains `PENDING`.

#### Background

Spec §13.4 enumerates the proposal lifecycle:

```
DRAFT → PENDING → ESCALATED → AUTO_APPROVED | EXTERNALLY_APPROVED | REJECTED | EXPIRED | SUPERSEDED
```

The audit's second pass proposed renaming the field `proposalLifecycleState` → `proposalStatus` and the value `PENDING` → `EMITTED`.

#### Why deferred

`PENDING` is the lifecycle state name in the frozen spec. Changing it requires coordinated edits across §13, all examples, all conformance vectors, and the artifact schema. The standalone proposal example (`examples/proposal-example.json`) previews the v0.2 direction with `proposalStatus: EMITTED`; the spec body and `core/proposal.schema.json` retain v0.1 naming.

#### V0.2 review questions

1. Is `proposalStatus` clearer than `proposalLifecycleState`? (Argument for: shorter, matches `decisionStatus`. Argument against: "status" is overloaded; "lifecycle state" is unambiguous about being a state-machine state.)
2. Is `EMITTED` clearer than `PENDING`? (Argument for: describes what happened, not what's awaited. Argument against: "PENDING" matches the broader pattern of "awaiting decision.")
3. Does any rename affect interoperability with implementations targeting v0.1?

#### Migration path if approved

- Add new field name as alias; deprecate old.
- Update spec, schema, examples, vectors.
- Conformance vectors test both names during transition.

### 3. T4.1 — Full Obligations Registry

**Status:** Tier 4 placeholder.
**Origin:** Tier 3 deferral.

#### Purpose

A registered, typed system of obligations attached to messages and proposals — commitments that must be discharged as part of the run's completion. Examples: "this proposal must be retained for 7 years," "this decision must trigger downstream notification," "this artifact must be cryptographically destroyed at retention end."

v0.1 carries `dataClasses` and `retentionClass` as ad-hoc fields on `ArtifactRef`. T4.1 would formalize obligations as first-class artifacts with their own discharge semantics.

#### Sketch

- `ObligationRef` reference subtype (sixth subtype joining the existing five).
- `Obligation` artifact: type, target, due-by, discharge-conditions, audit linkage.
- `ObligationDischargeRecord` artifact: evidence that the obligation was met.
- New gate step for obligation discharge tracking.

#### V0.2 review questions

1. What's the minimum useful obligation registry? (Retention obligations? Notification obligations? Both?)
2. How are unmet obligations escalated? (Cross-tenant audit? Runtime alert? `OBLIGATION_OVERDUE` decision artifact?)
3. Does this overlap with existing `policySnapshotId` mechanics?

### 4. T4.2 — Data Handling Block

**Status:** Tier 4 placeholder.
**Origin:** Tier 3 deferral.
**Now includes:** Consent / Purpose Limitation (folded in from prior standalone proposal).

#### Purpose

Promote data classification from `dataClasses` array on `ArtifactRef` to a structured envelope-level block carrying full data-handling requirements: classification, residency, retention, processing constraints, sharing constraints, redaction requirements, **consent and purpose-of-use**.

v0.1's `dataClasses` is a tag list. Real data governance needs structure — classification levels, source jurisdictions, processing-permission scopes, downstream-sharing rules, and consent-bound purpose limitation.

#### Sketch

New top-level `dataHandling` envelope block. Per-block fields:

- `classification` — sensitivity level enum.
- `residencyJurisdictions[]` — declared data residency requirements.
- `processingConstraints[]` — what may be done with the data.
- `sharingConstraints[]` — downstream sharing rules.
- `redactionProfile` — redaction requirements.
- `retentionPolicy` — retention class and discharge.
- **`purposeOfUse`** — declared purpose for which data is being processed in this run.
- **`allowedPurposes[]`** — purposes the data may be used for, derived from consent or policy.
- **`consentRef`** — reference to the consent record authorizing this processing.
- **`consentScope`** — scope of the consent (per-tenant, per-purpose, time-bounded, etc.).
- **`purposeViolation` behavior** — what happens when actual use diverges from declared purpose (reject? flag? escalate?).

Cross-tenant policies can reference `dataHandling` requirements as part of their authorization conditions. Auto-approval can be gated on consent freshness and purpose alignment.

#### Consent / purpose-limitation specifics

The folded-in concern: ensure data is used only for the declared purpose. Examples:

- Data collected for payroll cannot be used for performance scoring.
- Customer support transcripts cannot be used for marketing analysis.
- Candidate resume cannot be used outside hiring workflow.

Particularly important for GDPR, HR, healthcare, and regulated workflows. Consent is a sub-axis of data handling, not a separate protocol axis — keeping them together avoids two competing data-governance designs.

#### V0.2 review questions

1. Should `dataHandling` be on the envelope or on each `ArtifactRef`? (Or both, with envelope-level as default and per-artifact override?)
2. How does `dataHandling` interact with `crossTenantPolicy`?
3. Does this duplicate `dataClasses`? (Likely supersede — v0.1 would deprecate `dataClasses` over a 90-day window.)
4. Should `purposeOfUse` be free-text, enum, or registered? (Free-text is flexible but unenforceable; enum is enforceable but inflexible; registered purpose-codes are a middle ground.)
5. Should consent records be in-band (referenced from envelope) or out-of-band (cited but not protocol-defined)?
6. Should purpose violation always reject, or sometimes flag-and-continue with audit emission?

### 5. T4.3 — Tool Side-Effect Classes

**Status:** Tier 4 placeholder.
**Origin:** Tier 3 deferral. (v0.1 declares `sideEffectClass: READ_ONLY` on capability declarations but doesn't enforce other classes.)

#### Purpose

Promote side-effect classification from a declarative tag on capability manifests to a protocol-enforced envelope-level mechanism. The broker (§10) and authorization decision (§3a.6) consult side-effect class to determine policy gates.

#### Sketch

- Reserved `sideEffectClass` enum extended: `READ_ONLY`, `IDEMPOTENT_WRITE`, `NON_IDEMPOTENT_WRITE`, `EXTERNAL_SIDE_EFFECT`, `IRREVERSIBLE`.
- Each class triggers different broker-state behaviors (e.g., `IRREVERSIBLE` defaults to `APPROVAL_NEEDED`).
- New `sideEffectClass` field on tool-call envelopes (when not derivable from binding).

#### V0.2 review questions

1. Are five classes the right granularity?
2. Should `IRREVERSIBLE` always require human approval, or is policy-driven?
3. How does this interact with `idempotencyKey` semantics?

### 7. T4.5 — Concurrency Locks

**Status:** Tier 4 placeholder.
**Origin:** Tier 3 deferral.

#### Purpose

Specify protocol-level concurrency mechanisms to prevent conflicting parallel runs. v0.1 has implicit run-isolation via `runId` but no specified locking primitive for shared resources across runs.

#### Sketch

- `LockAcquireRequest` and `LockAcquireDecision` (analogous to broker decisions).
- Lock scope: tenant + resource identifier + lock kind (read/write/exclusive).
- TTL-bound; auto-released on expiry.
- Cross-runtime locks via shared lock authority (not specified by protocol).

#### V0.2 review questions

1. Distributed locks across runtimes — protocol concern or deployment concern?
2. Should locks be advisory or enforced?
3. How does this interact with T4.8's pre-execution gate (lock-availability could be a precondition)?

### 8. T4.6 — Budget Controls

**Status:** Tier 4 placeholder.
**Origin:** Tier 3 deferral.

#### Purpose

Specify protocol-level budget mechanisms beyond v0.1's individual-message gating. Budgets cover token spend, monetary cost, tool-call counts, runtime duration, per-run and per-tenant aggregation.

v0.1 mentions budget exhaustion as an error code (`BUDGET_EXHAUSTED`) but doesn't specify how budgets are scoped, allocated, depleted, or replenished.

#### Sketch

- `BudgetSnapshot` reference artifact.
- `BudgetReservation` and `BudgetReservationDecision`.
- Multi-tier budgets: per-message, per-run, per-conversation, per-tenant, per-time-window.
- Pre-execution budget reservation (links to T4.8).
- Budget exhaustion produces typed `BudgetExhaustedDecision`, not just an error.

#### V0.2 review questions

1. Should budgets be reservation-based (commit upfront) or consumption-based (track spend)?
2. How are mid-run budget exhaustions handled? (Cancel? Pause? Escalate?)
3. Does this require currency-aware budget types, or are abstract "budget units" sufficient?

### 9. T4.7 — Reasoning Trace Evidence & Reasoning-Time Risk Visibility

**Status:** Open for v0.2 review.
**Origin:** Post-audit design conversation.
**Locked decision for v0.1:** Out of scope. v0.1 does not address reasoning-time concerns.

#### Purpose

Make agent reasoning **observable and auditable** when policy requires it. Provide an attachment point in the protocol for runtime-captured reasoning traces, without requiring OACP to define how reasoning is conducted.

#### Core boundary

> **OACP does not filter token-level reasoning. OACP can require reasoning trace evidence after reasoning and before proposal/decision. Runtime/model layer owns real-time reasoning controls.**

Token-level filtering, logit bias, provider-specific decoding controls, chain-of-thought forcing, step-by-step reasoning envelope emission, model-internal behavior, and prompt-injection defense are **runtime/model-layer controls, not protocol controls**.

#### What it adds to OACP

| Element | Description |
|---|---|
| `REASONING_TRACE` artifact subtype | New `artifactType` value joining the existing reserved set (`DOCUMENT`, `EVIDENCE_REPORT`, etc.). |
| `ReasoningTraceRef` | An `ArtifactRef` profile constrained to `artifactType: REASONING_TRACE`. Tenant-scoped, hash-verified, retention-bound, redaction-aware. |
| `reasoningTraceRef` on proposal | Optional at `OACP-Operational`/`OACP-Governed`. Required at `OACP-Regulated` for capability-scoped proposals when policy demands. Populated by runtime, not by agent. |
| Reasoning-trace retention classes | New retention classes including `REASONING_TRACE_REDACTED` and `REASONING_TRACE_FULL`. |
| Reasoning-trace policy gates | A new auto-approval precondition (would extend §13.5 from 10 to 11): "Reasoning trace, if present and policy-required, is consultable and meets reasoning-trace policy gates." |
| Audit linkage | Audit chain entries reference reasoning trace where present. |

#### Lifecycle

```
Agent reasons
    ↓
Ojas runtime captures reasoning trace if policy requires
    ↓
Runtime stores trace as ArtifactRef with artifactType: REASONING_TRACE
    ↓
OACP proposal references reasoningTraceRef
    ↓
Policy / gates may inspect trace before approval or auto-approval
```

#### Threat-model addition

New surface (extending §26.3):

> **Reasoning-time leakage or drift.** An agent may process sensitive data, cross-tenant content, unsafe tool plans, or policy-violating intermediate reasoning before emitting a compliant OACP envelope.

**Mitigation:** `ReasoningTraceRef` as evidence; runtime-managed redaction; policy-controlled retention; review/approval gating; audit visibility.

**Residual risk:** OACP can make reasoning observable and auditable. It cannot guarantee internal reasoning was safe unless the runtime/model-serving layer enforces it. Adversarial models may reason silently or fabricate traces; the protocol doesn't prevent this.

#### Out of scope (even at v0.2)

- Token-level filtering during sampling.
- Step-by-step gate execution mid-reasoning.
- Forcing the model to reason in a particular way.
- Provider-specific model-serving controls.
- Real-time reasoning interruption.

#### V0.2 review questions

1. Should reasoning-trace evidence be optional or mandatory at `OACP-Regulated`?
2. What's the minimum trace fidelity that counts as evidence? (Final reasoning output? Step-by-step? Token-level?)
3. How are traces redacted before storage to avoid PII bloat?
4. Can a trace be cryptographically signed by the model serving layer? (Likely no; deferred to runtime.)

### 10. T4.8 — Pre-Execution Run-Intent Gate

**Status:** Open for v0.2 review.
**Origin:** Post-audit design conversation.
**Locked decision for v0.1:** Out of scope. v0.1 has no protocol-level pre-execution checkpoint.

#### Purpose

Decide whether a run should start **before** the agent spends tokens, resolves data, calls tools, or emits proposals. Provide a typed artifact + decision pattern for pre-execution review covering operational concerns (cost, freshness, conflict, residency), regulatory concerns (blackout windows, mandatory human review), and policy concerns (ethics, jurisdiction, scope).

#### What it adds to OACP

| Element | Description |
|---|---|
| `RUN_INTENT` artifact subtype | New `artifactType`. Carries: task description, input refs, expected outputs, anticipated tool calls, anticipated cost estimate, declared data classes, declared regulatory regimes, anticipated risk class, target binding. Tenant-scoped, hash-verified, immutable. |
| `RUN_INTENT_DECISION` decision subtype | New `decisionType`. Decision artifact emitted by the policy/review authority responsible for pre-execution review. |
| `runIntentRef` on `RUN_STARTED` | Required at `OACP-Governed`+ when pre-execution review is enabled by profile/policy. |
| `runIntentDecisionRef` on `RUN_STARTED` | Required at `OACP-Governed`+ when pre-execution review is enabled. The decision MUST have `result: APPROVED` for `RUN_STARTED` to proceed. |
| New gate step (call it 0.5) | "Verify run intent and run-intent decision." Runs before identity verification, because if the run was never approved, identity doesn't matter. |
| New profile capability `OACP-PreExecution-Review` | Additive capability profile. Mandates `RunIntent` + `RunIntentDecision` artifacts on every run. |

#### Decision results

```
APPROVED
REJECTED
MODIFIED
DEFERRED
EXPIRED
NEEDS_HUMAN_REVIEW
```

#### Reason codes

```
BUDGET_EXCEEDED
INPUT_UNAVAILABLE
INPUT_QUALITY_FAILED
BINDING_STALE
POLICY_BLACKOUT
CONFLICT_DETECTED
MANDATORY_HUMAN_REVIEW
INPUT_RESIDENCY_VIOLATION
CALIBRATION_STALE
WRONG_AGENT_FOR_TASK
ETHICS_RESTRICTION
REGULATORY_RESTRICTION
```

Plus namespace-prefixed custom codes per spec §2b.7.

#### Core rule

> **`RUN_STARTED` at `OACP-Governed`+ should require a valid approved `RunIntentDecision` when pre-execution review is enabled by profile/policy.**

#### Realistic enterprise scenarios this covers

- **Cost gating** — reject runs whose estimated token spend exceeds remaining budget.
- **Stale data** — reject runs targeting archived/cold-storage inputs that would incur retrieval cost.
- **Regulatory blackout** — reject trading-analytics runs during earnings blackout windows.
- **Stale credentials** — reject scheduled runs whose binding will be expired at execution time.
- **Input quality** — reject runs against malformed inputs that will produce garbage output.
- **Tenant context drift** — reject scheduled runs whose authorizing cross-tenant policy became `RETIRED` between scheduling and execution.
- **Concurrency conflicts** — reject runs that conflict with active runs holding contended locks.
- **Mandatory human authorization** — reject auto-execution for task classes requiring human pre-clearance (regulator communication, production billing rules, model deployment).
- **Calibration freshness** — reject runs using stale calibration relative to current operational conditions.
- **Cross-domain routing** — reject mismatched agent/task pairings before run starts.
- **Retry with stale state** — reject retries whose prior state has been invalidated by upstream changes.
- **Per-input residency** — reject runs whose input data has residency requirements incompatible with the configured tool set.
- **Ethics restriction** — reject runs deemed ethics-noncompliant by the configured ethics policy.

#### Threat-model addition

New surface (extending §26.3):

> **Adversarial intent declaration.** An agent or runtime declares benign intent in `RunIntent` while the actual execution diverges. Mitigation is detection (audit comparison of `RunIntent` against actual `RuntimeTrace` and emitted artifacts), not prevention. Multi-party review at `OACP-Regulated`.

#### How T4.7 and T4.8 work together

```
T4.8 asks:  Should this run start?
T4.7 asks:  What happened during reasoning, and can it be audited
            before proposal/approval?
```

Combined lifecycle:

```
Pre-execution gate (T4.8)
    ↓
Agent reasoning
    ↓
Reasoning trace evidence (T4.7)
    ↓
Proposal
    ↓
Review / decision
```

T4.8 brackets the run from the front. T4.7 brackets the reasoning step within the run. Together they bracket the protocol's blind spot in v0.1 — what happens between `RUN_STARTED` and the first emitted artifact.

#### V0.2 review questions

1. Should `RunIntent` declaration be required even when pre-execution review is disabled? (Argument for: declared intent is itself an audit anchor. Argument against: protocol overhead.)
2. How granular should `RunIntent` be? (One per run? One per task within a run?)
3. Should `RunIntentDecision` carry a budget/cost estimate that becomes a runtime budget at run start?
4. What happens if pre-execution review is enabled but the policy engine is unavailable? (Fail-closed by default? Profile-dependent?)

### 11. T4.9 — Capability Risk Hints

**Status:** Open for v0.2 review.
**Origin:** Post-audit design proposal (originally framed as "Capability Risk Scoring," renamed for clarity).

#### Purpose

Allow capability manifests to declare **risk hints** that the policy engine consults when deriving the per-proposal risk class (per §13.5). v0.1 already says risk class is derived by external authority from `agentId`, `taskType`, `outputType`, `domainPack`, trigger conditions, and validation evidence. T4.9 adds the capability manifest itself as an input to that derivation.

#### Why "hints" not "scoring"

T4.9 is **not** a pre-execution risk score — that's T4.8's job. T4.9 is **manifest-level metadata** that the existing risk-derivation step can consume. The framing matters: hints inform the policy engine's derivation; they don't replace it. Agents still cannot self-assign risk class.

#### Sketch

- New `riskHints` block on capability manifest:

  ```
  riskHints:
    intrinsicRisk: LOW | MEDIUM | HIGH | CRITICAL
    riskFactors[]:
      - factor: SIDE_EFFECT_IRREVERSIBLE
        weight: HIGH
      - factor: REGULATED_DOMAIN
        weight: MEDIUM
    contextSensitive: boolean
  ```

- Policy engine consumes these hints alongside other inputs when deriving `riskClass` on proposals.
- Hints are **advisory** — policy may override based on tenant-specific or domain-specific rules.
- Hints DO NOT bypass the existing rule that agents cannot assign their own risk class. They're declared on the manifest by the capability author and reviewed at manifest registration time.

#### V0.2 review questions

1. Should `intrinsicRisk` be a single enum or a per-context lookup?
2. Are five risk factors enough, or does each domain need its own?
3. How does this interact with T4.3 (Tool Side-Effect Classes)? (Likely complementary — side-effect class is one input to risk hints.)
4. Should hints be machine-derivable from manifest properties (auto-computed) or hand-declared (author-asserted)?

### 12. T4.10 — Policy Simulation / Dry-Run Mode

**Status:** Open for v0.2 review.
**Origin:** Post-audit design proposal.

#### Purpose

Provide a protocol-level mechanism to ask: "Would this message / run / tool call / proposal be allowed under the current policy?" — without actually executing anything, without side effects, without authoritative decisions being emitted, without budgets being consumed.

This is a high-value enterprise capability for testing policy changes, simulating cross-tenant flows, validating new manifests, pre-checking risky tool calls, debugging rejected messages, and onboarding new agents safely.

#### Sketch

- New `executionMode` field on the envelope: `EXECUTE` (default) | `SIMULATE` | `REPLAY`.
  - `SIMULATE` propagates through every gate; brokers refuse all `ALLOW` decisions and emit `WOULD_ALLOW` instead. No actual side effects occur.
  - `REPLAY` is the existing replay-mode behavior (see spec §3c).
- New artifact subtypes:
  - `POLICY_SIMULATION_REQUEST` — declares what would be evaluated.
  - `POLICY_SIMULATION_RESULT` — captures the would-have-been outcome at each gate.
- New broker decision values: `WOULD_ALLOW`, `WOULD_DENY`, `WOULD_SANITIZE`, `WOULD_QUARANTINE`, `WOULD_APPROVAL_NEEDED`.
- Simulation results are **non-authoritative**. They cannot become real decisions; they are for analysis only.
- Audit: simulations are logged but tagged `executionMode: SIMULATE` so they don't pollute the production audit chain.

#### Integrity guarantees

The hardest design constraint: a simulation **must not accidentally execute**. Mechanisms:

1. `executionMode: SIMULATE` propagates through every envelope from the original entry point — receivers can verify the chain.
2. Brokers that receive `SIMULATE` mode envelopes refuse to dispatch to actual tools; they consult policy and return synthetic results.
3. Simulation envelopes are routed to a separate trace stream from production envelopes.
4. Tenant scope is identical for simulation and production — there's no "simulation tenant" — so the simulation reflects real tenant policy.

#### V0.2 review questions

1. Should simulation be available in all profiles or only `OACP-Operational`+ (where policy is enforced)?
2. How is simulation cost accounted for? (Counts against budget? Free?)
3. Can simulation results be cached for repeated identical queries? (Probably yes with TTL.)
4. Should simulation support time-travel — "would this have been allowed yesterday under yesterday's policy"? (Probably yes via existing `policySnapshotId`.)

### 13. T4.11 — Output Content Safety Classification

**Status:** Open for v0.2 review.
**Origin:** Post-audit design proposal.

#### Purpose

Classify generated output content **before** it is delivered, displayed, or acted on. v0.1 has risk class (consequences of acting on output) but no protocol-level mechanism for declaring that the output content itself is safe / sensitive / regulated / disallowed.

The distinction matters: **risk class** describes consequences of action on the output; **safety classification** describes properties of the output's content. An output may be low-risk to act on but contain sensitive content; or high-risk to act on but contain plain content.

#### Sketch

- New `OUTPUT_SAFETY_CLASSIFICATION` artifact subtype, or structured field on proposals: `safetyClass`.
- Reserved values: `SAFE`, `NEEDS_REVIEW`, `SENSITIVE`, `REGULATED`, `DISALLOWED`, `HIGH_IMPACT`.
- Sits between proposal emission and decision; can be a precondition to auto-approval (extending §13.5 preconditions).
- Classification is performed by a registered classifier (manifest declares the classifier, just like other gates).

#### Distinction from risk class

| Dimension | What it describes | Example |
|---|---|---|
| `riskClass` (v0.1) | Consequence of *acting* on the output | Updating a billing record = HIGH risk regardless of content |
| `safetyClass` (T4.11) | Properties of the output *content itself* | Output mentions specific medical advice = REGULATED content |

Both can be true simultaneously; they're orthogonal axes.

#### Use cases

- Email content classification before send.
- Legal summary classification before delivery to client.
- Financial recommendation classification before acting.
- Medical draft classification before submission.
- HR hiring output classification before candidate communication.
- Customer-facing content classification before publication.

#### V0.2 review questions

1. Are six reserved classes the right granularity?
2. Should `DISALLOWED` always reject, or sometimes flag-and-quarantine?
3. How does this interact with T4.7 (Reasoning Trace Evidence)? (Reasoning trace is input; safety classification is output.)
4. Is the classifier itself a domain-pack concern or a protocol-level concern? (Probably domain-pack with protocol-level declared interface.)

### 14. T4.12 — Evidence Quality Scoring

**Status:** Open for v0.2 review.
**Origin:** Post-audit design proposal.

#### Purpose

Score whether the evidence supporting a proposal or decision is **strong enough** to support auto-approval. v0.1 has `EvidenceReport` artifacts but no protocol-level quality dimension. T4.12 extends `EvidenceReport` with a structured quality metric.

#### Sketch

New structured field on `EvidenceReport`:

```
evidenceQuality:
  overallScore: 0.0–1.0
  dimensions:
    completeness:        0.0–1.0
    sourceReliability:   0.0–1.0
    freshness:           0.0–1.0
    validationPassRate:  0.0–1.0
    conflictCount:       integer
    humanReviewedFlag:   boolean
  scoringMethod: string
  scoringPolicySnapshotId: string
```

New auto-approval precondition (extends §13.5 to 11 preconditions): "Evidence quality score meets the policy threshold."

#### Why scoring vs. binary pass/fail

A binary "evidence passed" / "evidence failed" obscures the strength axis. Two evidence reports both passing all gates may still differ substantially in completeness and reliability. Auto-approval policy should be able to require not just "evidence is valid" but "evidence is *strong*."

#### V0.2 review questions

1. Should the scoring method be protocol-defined or registered-and-versioned per deployment?
2. Should weak evidence trigger automatic external review even when other preconditions pass?
3. How are evidence quality scores composed across multi-source evidence?
4. Does this duplicate calibrated confidence on the proposal? (No — confidence is about the proposal's correctness; quality is about the evidence's strength.)

### 15. T4.13 — Kill-Switch / Emergency Stop Semantics

**Status:** Open for v0.2 review.
**Origin:** Post-audit design proposal.

#### Purpose

Provide protocol-level emergency-stop semantics for production operations and incident response. v0.1 has no specified mechanism for "stop everything immediately" — deployments rely on out-of-band runtime mechanisms which lack audit trail and cross-runtime coordination.

This is overdue. Any deployment claiming `OACP-Regulated` will need this.

#### Sketch

- New artifact subtypes: `KILL_SWITCH_REQUEST`, `KILL_SWITCH_DECISION`, `KILL_SWITCH_RESUME_REQUEST`, `KILL_SWITCH_RESUME_DECISION`.
- Kill-switch levels (cumulative — higher levels include lower):

  ```
  STOP_RUN              — stop one specific run
  SUSPEND_AGENT         — stop all runs of one agent
  SUSPEND_CAPABILITY    — stop all use of one capability across agents
  SUSPEND_TENANT        — stop all runs in one tenant
  SUSPEND_TOOL          — stop all calls to one tool
  READ_ONLY_RUNTIME     — runtime accepts only READ_ONLY side-effect-class operations
  GLOBAL_EMERGENCY_STOP — runtime stops accepting any new work
  ```

- Authority asymmetry: **resuming requires higher authority than stopping**. Stopping at level N can be authorized by an `OACP-Governed` operator; resuming may require multi-party authorization at `OACP-Regulated`.
- Audit trail mandatory. Every kill-switch transition emits an `ApprovalDecision`-shaped record.
- TTL discipline: kill-switch states are time-bound by default; permanent stops require explicit non-expiring authorization.
- Cross-runtime coordination: not specified by protocol (deployment concern).

#### Interaction with existing v0.1

- `BLOCKED` capability lifecycle state (per §27.6) is the per-capability version of `SUSPEND_CAPABILITY`. T4.13 generalizes this to all the levels above.
- Existing `EXPIRED` and `REJECTED` proposal states remain unchanged. Kill-switch is orthogonal — it can fire at any point.
- Existing `replayAuthorizationRef` (§3c) does NOT bypass active kill-switch states.

#### V0.2 review questions

1. Should `GLOBAL_EMERGENCY_STOP` be reachable from inside the runtime, or only from external authority?
2. What's the recovery procedure after `GLOBAL_EMERGENCY_STOP`? (Multi-party resume? Manifest-level re-approval?)
3. How does kill-switch interact with in-flight proposals awaiting external review?
4. Should the kill-switch state itself be tenant-scoped (per-tenant kills) or runtime-global (one kill stops everyone)?

### 16. T4.14 — Trajectory-Level Authorization & Session Risk Memory

**Status:** Open for v0.2 review.
**Origin:** Post-audit design proposal (added after the initial 15-candidate queue was sealed).
**Locked decision for v0.1:** Out of scope. v0.1 evaluates messages and actions individually; it has no protocol-level mechanism for evaluating accumulated risk across a session or conversation trajectory.

#### Purpose

Extend OACP beyond per-message and per-tool-call governance by evaluating whether a sequence of individually valid actions is becoming risky over time.

#### Motivation

OACP v0.1 provides spatial governance: identity, authority, capability binding, authorization decisions, tool-broker state, tenant scope, replay, and audit are checked for each message or action. This does not fully address temporal risk, where a session or conversation accumulates risk through repeated low-risk actions, gradual privilege expansion, staged data movement, or drift from the approved run intent.

The distinction matters: a tool call that's individually allowed under all v0.1 gates may be the tenth in a sequence whose pattern indicates exfiltration. v0.1 has no primitive that sees "the tenth in a sequence" — every gate evaluates one envelope in isolation.

#### Default scope

The default scope for trajectory analysis is `conversationId`, because risk may accumulate across multiple runs in the same conversation. v0.2 review should decide whether a narrower `runId`-scoped mode is also required (for long-running single-run agents that exhibit within-run drift).

#### Candidate objects

- `SessionRiskState` — accumulated state across the session, updated as new actions are evaluated.
- `TrajectoryRiskScore` — quantified risk score for the current trajectory.
- `TrajectoryAuthorizationDecision` — typed decision artifact, same shape as existing `DecisionRef`.
- `trajectoryAuthorizationRef` — reference field on envelopes when trajectory authorization is enabled.

#### Candidate decisions

- `ALLOW_CONTINUE` — trajectory remains within risk envelope.
- `REQUIRE_REVIEW` — trajectory has crossed a threshold that requires external review before further action.
- `SUSPEND_SESSION` — trajectory is suspended; in-flight actions in the conversation are paused.
- `SUSPEND_CAPABILITY` — specific capability is suspended within the session.
- `QUARANTINE_SESSION` — session is moved to quarantine for forensic analysis.
- `TERMINATE_RUN` — current run is terminated; conversation may continue under stricter rules.

#### Relationship to other v0.2 candidates

This candidate is the temporal-governance counterpart to several existing spatial v0.2 candidates:

- **T4.6 Budget Controls** — also tracks accumulating session/run state, but for resource consumption rather than risk accumulation.
- **T4.7 Reasoning Trace Evidence** — provides per-run reasoning evidence; T4.14 provides per-trajectory risk evidence.
- **T4.8 Pre-Execution Run-Intent Gate** — gates run start; T4.14 gates run continuation.
- **T4.13 Kill-Switch / Emergency Stop Semantics** — provides operator-driven termination; T4.14 provides policy-driven trajectory-based termination.

v0.2 review must decide whether trajectory authorization is a standalone primitive or a composition over T4.6 + T4.7 + T4.8 + T4.13. Both options are defensible; the design question is whether session-level risk is fundamentally distinct from these other axes or whether it emerges naturally when they're composed.

#### Framing

Trajectory authorization is **temporal governance** — a complement to v0.1's spatial governance, not a replacement. v0.1 asks "is this action allowed right now?" T4.14 asks "is this sequence of actions becoming dangerous over time?" Both questions need answering for governed agent operation.

#### V0.2 review questions

1. Should `conversationId` be the default scope, or should `runId` and `tenantId` also have defined trajectory-analysis modes?
2. Is `SessionRiskState` a single artifact that mutates, or an append-only sequence of state snapshots?
3. How does `TrajectoryRiskScore` interact with the `riskClass` derivation for individual proposals (per §13.5)?
4. Should `TERMINATE_RUN` decisions be reversible (resumable run) or always terminal?
5. Is trajectory authorization standalone, or composition over T4.6 + T4.7 + T4.8 + T4.13?
6. How does trajectory authorization interact with replay (§3c)? Does replaying a session reproduce its risk score?
7. What's the minimum trajectory length before risk evaluation activates? (One action is too few; ten may be too many.)

#### Scope

This is a v0.2 candidate. It does not modify OACP v0.1.

## Reframing Required

Items in this section have **real underlying concerns** but their original framing has protocol-shape problems that prevent them from entering the accepted v0.2 queue as-is. Tracked here so the concerns aren't lost; they need redesign before they can become candidates.

---

### A. Runtime Health / Trust Posture

**Original framing:** A runtime declares its own health status (`HEALTHY`, `DEGRADED`, `POLICY_CACHE_STALE`, `KEY_ROTATION_PENDING`, `AUDIT_BACKLOGGED`, `CONFORMANCE_FAILED`, `READ_ONLY_MODE`, `SUSPENDED`) inside OACP envelopes.

#### Why this fails as written

**Self-attestation has trust circularity.** A compromised runtime would always self-attest as `HEALTHY`. Building a protocol field that the runtime fills in about its own health is the opposite of what a regulator-grade governance protocol should do — it lets the worst case lie cleanly. The whole point of an envelope being signed by a runtime is that the runtime is *the thing being verified*, not the verifier.

#### Required reframing

`RuntimeHealthAttestation` as an artifact subtype emitted by an **external monitoring/attestation authority**, not by the runtime itself.

- `RUNTIME_HEALTH_ATTESTATION` artifact subtype.
- Externally signed by a registered attestor (deployment-specific — could be an SRE platform, a regulator-mandated attestor, a third-party monitoring system).
- Optional reference on envelopes (e.g., `runtimeHealthRef` on `RUN_STARTED`) for runtimes that participate in attested-health flows.
- Aegis-style policy can consume the attestation as one input among many when deciding auto-approval.
- Attestation has its own freshness rules — stale attestations don't authorize work.

#### Why this still belongs in the queue (as reframed)

Attested runtime health is genuinely useful for `OACP-Regulated` deployments. The reframed version is small-surface, follows the existing `Reference` pattern, and doesn't have the trust-circularity problem.

**Disposition:** Defer to v0.3 / Tier 5 in current form. Listed below as D5.

---

### B. Agent Drift Detection

**Original framing:** OACP detects when an agent's behavior has drifted from a baseline (sudden confidence shifts, higher rejection rates, more review requests, more policy violations, different tool-selection patterns, more hallucinated evidence references).

#### Why this fails as written

**Statistical drift detection is an observability concern, not a protocol concern.** Detecting drift requires:

- A baseline of historical behavior over time.
- Statistical comparison (rolling windows, control charts, distribution shifts).
- Out-of-band alerting.
- Threshold tuning per agent and per deployment.

None of that fits inside an OACP envelope. The protocol's role is to make drift *observable* (which it does today via `RuntimeTrace`, `EvidenceReport`, audit chains, and decision records). Detecting drift is a deployment-side analytics function consuming those artifacts.

#### Required reframing

`AgentDriftAlert` as an artifact subtype emitted by an **external drift detector**, not produced inside the runtime as part of normal envelope flow.

- `AGENT_DRIFT_ALERT` artifact subtype.
- Externally signed by a registered drift detector.
- Carries: detector identifier, agent under observation, baseline window, comparison window, drift dimensions, severity, evidence references back to the audit artifacts that informed the detection.
- Aegis-style policy can consume the alert as input to decisions about whether to continue auto-approval for the affected agent.
- Likely folds into the same `RuntimeHealthAttestation` pattern — drift alerts are a sub-class of external attestations about runtime/agent state.

#### Why this still belongs in the queue (as reframed)

Drift-aware governance is real and important. The reframed version is small-surface and reuses the external-attestation pattern.

**Disposition:** Defer to v0.3 / Tier 5 in current form. Listed below as D6.
